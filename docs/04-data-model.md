# Sonoscope — Data Model

One SQLite database (`library.db`) lives in the library root folder. All tables are in this file. WAL journal mode is enabled.

---

## Entity overview

```
dimensions ──┐
             ├── dimension_values
             │
samples ─────┼── tags ──────────────── dimension_values
             │        └─── dimensions
             │
             └── file_operations ─── operation_batches

organisation_presets   (standalone)
library_meta           (standalone, single row)
```

---

## Tables

### `library_meta`
Single-row table. Stores library-level state.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | always 1 |
| `root_path` | TEXT NOT NULL | absolute path to the library root folder |
| `created_at` | INTEGER NOT NULL | unix timestamp |
| `last_discovered_at` | INTEGER | timestamp of last completed discovery scan |

---

### `samples`
One row per audio file discovered in the library.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `path` | TEXT NOT NULL UNIQUE | absolute path on disk |
| `filename` | TEXT NOT NULL | basename, used for display and filename search |
| `relative_path` | TEXT NOT NULL | path relative to library root, for display |
| `format` | TEXT | wav, aiff, flac, mp3, ogg, … |
| `size_bytes` | INTEGER | |
| `duration_ms` | INTEGER | |
| `sample_rate` | INTEGER | Hz |
| `bit_depth` | INTEGER | null for lossy formats |
| `channels` | INTEGER | 1 = mono, 2 = stereo |
| `file_hash` | TEXT | fast hash (xxHash or similar) of file content; used during re-scan to distinguish moves from new files |
| `analysis_status` | TEXT NOT NULL | `pending` \| `analysing` \| `done` \| `failed` |
| `discovered_at` | INTEGER NOT NULL | unix timestamp |
| `last_seen_at` | INTEGER NOT NULL | updated on each discovery scan |
| `waveform_data` | BLOB | nullable; compact float32 array (~400 points) of normalised amplitude values, generated by the sidecar and cached here; populated on first playback |

**Notes:**
- `analysis_status = analysing` means the file was mid-analysis when the app last ran. On startup, any rows in `analysing` state are reset to `pending` to ensure they are re-queued.
- `file_hash` is computed during discovery. A cheap non-cryptographic hash (xxHash128) is used to keep discovery fast on large libraries.

---

### `dimensions`
Defines the classification axes available in the library.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `name` | TEXT NOT NULL UNIQUE | e.g. "Type", "Instrument", "Key" |
| `value_type` | TEXT NOT NULL | `enum` \| `multi_enum` \| `numeric` \| `text` |
| `is_system` | INTEGER NOT NULL | 1 = built-in, cannot be deleted |
| `sort_order` | INTEGER NOT NULL | display order in the UI |

**System dimensions (seeded at library creation):**

| name | value_type |
|---|---|
| Type | enum |
| Instrument | multi_enum |
| Key | enum |
| Mode | enum |
| Tempo | numeric |
| Mood | multi_enum |

---

### `dimension_values`
Allowed values for `enum` and `multi_enum` dimensions. Not used for `numeric` or `text` dimensions.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `dimension_id` | INTEGER NOT NULL | FK → dimensions |
| `value` | TEXT NOT NULL | e.g. "kick", "loop", "C#" |
| `is_system` | INTEGER NOT NULL | 1 = built-in, cannot be deleted while any tag references it |
| UNIQUE | | `(dimension_id, value)` |

---

### `tags`
Tag assignments — one row per (sample, dimension, value, source) combination. Multiple rows per sample per dimension are allowed (multi-value tags and multiple sources).

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `sample_id` | INTEGER NOT NULL | FK → samples |
| `dimension_id` | INTEGER NOT NULL | FK → dimensions |
| `value_id` | INTEGER | FK → dimension_values; NULL for numeric/text dimensions |
| `numeric_value` | REAL | used when dimension value_type = numeric |
| `text_value` | TEXT | used when dimension value_type = text |
| `source` | TEXT NOT NULL | `heuristic` \| `metadata` \| `model` \| `user` |
| `confidence` | REAL | 0.0–1.0; NULL for user-source tags |
| `created_at` | INTEGER NOT NULL | unix timestamp |
| `is_primary` | INTEGER NOT NULL DEFAULT 0 | primary value for display, sorting, and organisation |
| UNIQUE | | `(sample_id, dimension_id, value_id, source)` |

**Tag resolution rules:**
- All tags (from all sources) are stored and preserved.
- Each sample/dimension may have one primary tag used for display, sorting, and organisation.
- Analysis marks the highest-confidence auto tag for each dimension as primary.
- User selection marks the chosen value as primary without deleting other auto-source tags.
- If the user deletes their tag, an auto-source primary is restored.

**Conflict detection (computed, not stored):**
Conflicts are reserved for single-value dimensions where competing auto-source values are mutually exclusive. Multi-value dimensions such as `Instrument` can validly have multiple values, so additional values are alternatives rather than conflicts.

---

### `operation_batches`
One row per user-initiated reorganise action. Groups the individual file operations for rollback purposes.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `created_at` | INTEGER NOT NULL | unix timestamp |
| `pattern` | TEXT NOT NULL | the organisation pattern used, e.g. `{Type}/{Instrument}` |
| `mode` | TEXT NOT NULL | `move` \| `copy` |
| `file_count` | INTEGER NOT NULL | total files in this batch |
| `status` | TEXT NOT NULL | `completed` \| `rolled_back` |

---

### `file_operations`
One row per file touched in a batch. Enables per-batch rollback.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `batch_id` | INTEGER NOT NULL | FK → operation_batches |
| `sample_id` | INTEGER NOT NULL | FK → samples |
| `operation_type` | TEXT NOT NULL | `move` \| `copy` |
| `original_path` | TEXT NOT NULL | absolute path before the operation |
| `new_path` | TEXT NOT NULL | absolute path after the operation |
| `executed_at` | INTEGER NOT NULL | unix timestamp |

**Rollback:** iterating all `file_operations` for a batch and reversing `new_path → original_path`. Only applicable to `move` operations. After rollback the batch `status` is set to `rolled_back`; the sample `path` column is updated to `original_path`.

---

### `organisation_presets`
Saved organisation pattern presets.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK | |
| `name` | TEXT NOT NULL UNIQUE | e.g. "Type / Instrument" |
| `pattern` | TEXT NOT NULL | e.g. `{Type}/{Instrument}` |
| `is_system` | INTEGER NOT NULL | 1 = ships with the app, deletable by user |
| `created_at` | INTEGER NOT NULL | unix timestamp |

---

## Indexes

| Table | Index | Reason |
|---|---|---|
| `samples` | `path` | lookup by path during re-scan |
| `samples` | `file_hash` | move detection during re-scan |
| `samples` | `analysis_status` | queue pending files on startup |
| `samples` | `filename` | filename search |
| `tags` | `(sample_id, dimension_id)` | fetching all tags for a sample, conflict detection |
| `tags` | `(dimension_id, value_id)` | filtering file list by tag |
| `file_operations` | `batch_id` | rollback lookup |

## Seed data

On library creation the following rows are inserted:

**dimensions + dimension_values (system):**
- Type: `loop`, `one-shot`, `fill`, `break`, `texture`
- Instrument: `kick`, `snare`, `hi-hat`, `clap`, `cymbal`, `percussion`, `tops`, `drums`, `bass`, `chord`, `pad`, `synth`, `lead`, `vocal`, `fx`, `foley`, `guitar`, `piano`, `brass`, `woodwind`, `strings`
- Key: chromatic notes A–G with sharps/flats
- Mode: `major`, `minor`

`drums` is the full-kit value: it covers drum loops, breaks, and kit grooves that are not a single drum hit, so `{Instrument}` patterns have a home for them.

**organisation_presets (system):**
- "Type / Instrument" → `{Type}/{Instrument}`
- "Instrument / Type" → `{Instrument}/{Type}`
